Verdict first: this is a genuinely strong, build-ready set — faithful to the design, disciplined about cross-referencing (decisions only in the README, one fact per file, all DDL in 03, env vars in 05), and honest where it should be (the "verify, don't assume" flags, the consolidated open-questions register with owner columns, the compliance blocker called out as blocking rather than buried). The price-safety chain in particular is airtight end to end: tokens in the KB → `Render` after the model → a must-pass price-safety test → the golden-set publish gate. If I were signing off, I'd sign off on the structure. The critique below is the next layer — real issues an expert would catch, not structural problems.

**The one decision to reconcile before anything else: the window.** Every file commits to Decision 9's "**window — last ~15 messages**" — README, 02 (context-on-read, the memory table, and the worked example's "window held only the last ~15 messages"), 06's reads, and the glossary. But the design conversation concluded somewhere else: *send the whole conversation* (with a generous cap as a safety valve), no window, no summary, because at this scale the chats are small and sending everything is simpler and higher-quality. The docs encode the older choice. They're internally consistent, so it's not a contradiction *within* the set — it's a divergence from where you landed, and because "~15 window" is threaded through five files plus the glossary, changing it later is multi-file surgery. Decide now, and note the tradeoff in Decision 9: the window is more robust if conversations ever get long (bounded prompt); whole-chat-plus-cap is simpler and loses nothing at 100 leads. Either is fine, but the set should say one thing.

**Idempotency needs state the "stateless, no-DB" brain doesn't have.** 06 and 01 say "keep the handler idempotent on the Chatwoot message id," which is the right instinct for webhook redelivery. But with no datastore, the brain can't remember which message ids it has already processed — certainly not across a restart. An in-memory TTL set covers the common case (Chatwoot retrying within seconds), but a duplicate spanning a deploy slips through. The docs assert idempotency without naming the mechanism, and it sits in mild tension with Decision 2. The honest resolution is cheap and worth writing down: an in-memory TTL dedup cache, and an explicit acceptance that a duplicate that escapes it produces at most a duplicate *private note* — harmless, because a human reviews before sending. State that, rather than implying true idempotency.

**Concurrent profile merges can silently lose a field.** 06 correctly documents that `MergeContactAttributes` is a read-modify-write (GET → merge → PUT) because the PUT replaces the whole `custom_attributes` object — and the same is true of the labels POST. What's missing is the concurrency case: two messages on the *same contact* close together (a customer firing two messages, or a webhook plus its retry both processing) both read the same attributes and both PUT, and the second clobbers the first's merge. That directly violates the "additive merge, never null a known field" guarantee from Decision 9 — not by nulling, but by losing a just-added field. Rare at 100 leads, but it's a real hole. The fix is a per-conversation (or per-contact) serialization — a keyed mutex or a tiny work queue so one contact's messages process in order. It's not mentioned anywhere; it should be a line in 02 or 06.

**`suggested_callback` is computed and then dropped.** It's in the `Draft` struct, in the JSON contract, and the worked example shows the model returning it — but the post-processing pipeline in 02 (steps 1–7) never does anything with it, and 09 has the rep snooze manually. So it's dead output: the model spends tokens producing a follow-up suggestion that never reaches the rep. Either cut it from the contract for v1, or add a pipeline step that surfaces it in the private note ("suggested follow-up: …") so the rep can act on it with a Chatwoot snooze. As written it's defined but unwired.

**The copy-paste friction is a bigger product risk than its footnote suggests.** 09 honestly notes that a private-note draft must be *copied into the composer* to send, because Chatwoot won't let a bot pre-fill an agent's reply box — and defers the "use this reply" button to Phase 3. But KPI #1 is *median first-response time dropping sharply*, and that's the whole product thesis. If every reply is read-note → select → copy → paste → send, the time saved over just typing a short answer shrinks, and reps quietly stop using the drafts. I'd treat this as a top-tier v1 risk, not an aside: validate the real copy-flow time during Phase 2 with an actual rep, and seriously consider pulling the Chatwoot dashboard-app "use this reply" button *forward* from Phase 3 — a dashboard app can inject text into the composer via Chatwoot's frontend SDK, which is exactly the missing affordance. Worth verifying that capability early, because adoption hinges on it more than on the model's answer quality.

**A smaller seam: `ChatwootReader.Profile(ctx, chatID)` needs the contact id, not the conversation id.** The profile lives on the contact, but `chatID` is the conversation; 06's mapping resolves it via `conversation.meta.sender.id` from the webhook payload. So the port signature (chatID only) doesn't carry the data the call needs, which means an unstated extra conversation→contact lookup or a `ChatID` type that bundles the contact id. It's loose rather than wrong — clarify in 02 whether the handler passes the contact id alongside, or `Profile` resolves it internally. (Open question #4 touches the contact_id path but not this signature gap.)

**Two refinements rather than gaps.** On auto-send (Decision 6, open Q9): the gate is the model's *self-rated* `confidence`, and LLM self-confidence is poorly calibrated and easily over-stated. 02 already requires the objective checks too (no dropped refs, no leftover tokens, not escalated), which is the right instinct — I'd just add that `confidence` should be calibrated against the *draft-acceptance rate* from KPIs before it's trusted as a threshold, and treated as one input, not the sole gate. And on compliance (the #1 blocker): if you choose PII minimization/redaction as the lawful-basis answer, that's a step *inside* `HandleMessage` (redact before the `Drafter` call), so I'd design the redaction seam into the pipeline now — even as a no-op — rather than retrofit it after the legal decision; retrofitting a pipeline stage is more disruptive than reserving the slot.

**Cosmetic, but worth a pass:** the example media refs drift across files — 02's worked example uses `pricing_table` and `add_cashier_video`, while 03 and 08 use `tariffs_infographic_ru`, and 06's Playground gives `add_cashier_video` the kind `screen_recording`. None of it is load-bearing, but the canonical example reads as three different catalogs; pick one ref set and use it everywhere so a reader following the thread isn't second-guessing. (Also minor: `Draft.SuggestedStatus` is a `string` while the contract's `suggested_status` is an object `{stage}` — the flattening is fine but 02 should note it.)
