## Repo layout

Hexagonal, mirroring the main repo's package roles. The core (`domain`, `usecase/assistant`) has no infrastructure imports; adapters live under `infrastructure/`; transport under `ports/http/`.

```
xpayment-crm/
  cmd/main.go                 # build the container, start HTTP server, graceful shutdown
  internal/
    domain/                   # Draft, Message, ChatID, ResolvedAsset, AssistantConfig ‚Ä¶ (stdlib only)
    usecase/assistant/        # HandleMessage(ctx, chatID, inbound) ‚Üí Draft  + the ports (Decision 10)
      ports.go                # ChatwootReader, ChatwootWriter, Drafter, KnowledgeRetriever, ConfigStore, Prices
      service.go              # the pipeline (02-assistant-brain.md)
      mocks/                  # generated by mockery
    infrastructure/
      chatwoot/               # ChatwootReader + ChatwootWriter (REST adapter, 06)
      anthropic/              # Drafter (LLM adapter, prompt caching)
      knowledge/              # KnowledgeRetriever (loads KB + media catalog)
      pricebook/              # Prices.Render (token ‚Üí number)
      postgres/               # repos: assistant_configs, kb_topics, kb_assets, tariffs, placeholders
      config/                 # Config struct + getEnv loader (05)
    ports/http/
      router.go               # chi router + middleware
      webhook.go              # POST /v1/assistant/webhook/chatwoot  (06)
      admin.go                # admin API (config/KB/media/prices/playground) (06, 08)
  migrations/                 # NNNN_*.sql + embed.go  (goose, auto-run on startup)
  docs/                       # this set
  Dockerfile  docker-compose.yml  Makefile  .env.example  .mockery.yml
```

Reuse the package-role split from `xpayment/internal/` (`domain` ‚Üí `usecase` ‚Üí `infrastructure` / `ports`) and the lazy-init DI container from `xpayment/providers/container.go`.

## Dockerfile

Same multi-stage pattern as `xpayment/Dockerfile`: build a static binary on `golang:1.25-alpine`, run it on a minimal `alpine` with a non-root user. Migrations are **embedded in the binary** (see [Migrations](#migrations)), so the runtime image carries no SQL files.

```dockerfile
# Build stage
FROM golang:1.25-alpine AS build
WORKDIR /app
RUN apk add --no-cache git
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main ./cmd/main.go

# Deploy stage
FROM alpine:3.19 AS deploy
RUN apk add --no-cache tzdata ca-certificates wget   # wget is used by the healthcheck
RUN adduser -S -u 1001 appuser
ENV TZ=Asia/Almaty
WORKDIR /app
COPY --from=build /app/main ./main
EXPOSE 8080
USER appuser
ENTRYPOINT ["./main"]
```

Note: the brain connects to its Postgres over **TCP** (its own DB container), so it does not need the `GID 70` postgres-socket trick the main repo uses for a host socket. Otherwise identical.

## Local stack (docker-compose)

This is the one genuinely new piece: the main repo's compose runs *one* product; the brain's local stack must stand up **three** (Chatwoot, Evolution, the brain) plus a tunnel so Chatwoot can reach the brain's webhook from your machine.

Chatwoot and Evolution publish their own official compose files ‚Äî **compose them in rather than hand-rolling**. Sketch of the combined topology:

```yaml
# illustrative ‚Äî pin image versions, set secrets via .env, verify upstream envs
services:
  # ‚îÄ‚îÄ Chatwoot (hub) ‚îÄ‚îÄ its own Postgres + Redis ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  chatwoot-postgres: { image: postgres:16-alpine, volumes: [chatwoot_pg:/var/lib/postgresql/data] }
  chatwoot-redis:    { image: redis:7-alpine }
  chatwoot:
    image: chatwoot/chatwoot:latest        # pin a real tag
    depends_on: [chatwoot-postgres, chatwoot-redis]
    env_file: [.env]                        # POSTGRES_*, REDIS_URL, SECRET_KEY_BASE, FRONTEND_URL ‚Ä¶
    ports: ["3000:3000"]

  # ‚îÄ‚îÄ Evolution (WhatsApp gateway) ‚îÄ‚îÄ its own Postgres + Redis ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  evolution-postgres: { image: postgres:16-alpine, volumes: [evo_pg:/var/lib/postgresql/data] }
  evolution-redis:    { image: redis:7-alpine }
  evolution:
    image: atendai/evolution-api:latest     # pin a real tag
    depends_on: [evolution-postgres, evolution-redis]
    env_file: [.env]                        # AUTHENTICATION_API_KEY, CHATWOOT_* integration ‚Ä¶
    ports: ["8081:8080"]

  # ‚îÄ‚îÄ Brain (this repo) ‚îÄ‚îÄ its own Postgres ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  brain-postgres:
    image: postgres:16-alpine
    environment: { POSTGRES_USER: brain, POSTGRES_PASSWORD: brain, POSTGRES_DB: brain_db }
    volumes: [brain_pg:/var/lib/postgresql/data]
    healthcheck: { test: ["CMD-SHELL", "pg_isready -U brain -d brain_db"], interval: 5s, retries: 10 }
  brain:
    build: { context: ., dockerfile: Dockerfile }
    env_file: [.env]                        # see 05-configuration.md
    depends_on: { brain-postgres: { condition: service_healthy } }
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "-O", "/dev/null", "http://localhost:8080/health"]
      interval: 15s; timeout: 3s; start_period: 10s; retries: 3
    restart: unless-stopped
    stop_grace_period: 25s                   # graceful drain headroom (see Startup & shutdown)

  # ‚îÄ‚îÄ Webhook tunnel (local dev only) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  # Chatwoot must reach the brain's account webhook over a public HTTPS URL.
  tunnel:
    image: cloudflare/cloudflared:latest     # or ngrok
    command: tunnel --url http://brain:8080
volumes: { chatwoot_pg: , evo_pg: , brain_pg: }
```

Reuse from `xpayment/docker-compose.yml`: the `env_file: [.env]` wiring, the `wget /health` healthcheck, `restart: unless-stopped`, and `stop_grace_period: 25s`. The `x-app-base` YAML anchor pattern is worth copying if you add blue/green later.

**Local-dev wiring order:** bring up Chatwoot ‚Üí create the account + agent token ‚Üí bring up Evolution and configure its Chatwoot integration ([01 ¬∑ Evolution ‚Üî Chatwoot](01-infrastructure.md#1-evolution--chatwoot)) ‚Üí bring up the brain ‚Üí set the **tunnel URL** as Chatwoot's account webhook ([01 ¬∑ The two webhook kinds](01-infrastructure.md#2-the-two-webhook-kinds--do-not-conflate)).

## Migrations

Goose, embedded in the binary, **run automatically on startup** ‚Äî copy `xpayment/internal/infrastructure/postgres/postgres.go` and `xpayment/migrations/embed.go` verbatim in shape:

```go
//go:embed *.sql
var migrationsFS embed.FS            // migrations/embed.go

// infrastructure/postgres/postgres.go
func New(dsn string, migrationsFS embed.FS) (*sql.DB, error) {
    db, err := otelsql.Open("pgx", dsn, /* ‚Ä¶ otel span options ‚Ä¶ */)
    // ‚Ä¶
    goose.SetBaseFS(migrationsFS)
    goose.SetDialect("postgres")
    if err := goose.Up(db, "."); err != nil {       // applies pending migrations at boot
        return nil, fmt.Errorf("goose up: %w", err)
    }
    return db, nil
}
```

The brain's migrations create only its own tables (`assistant_configs`, `kb_topics`, `kb_assets`, `tariffs`, `placeholders` ‚Äî DDL in [03-content-and-data.md](03-content-and-data.md)). No conversation tables (Decision 2).

## Startup & shutdown

Mirror `xpayment/cmd/main.go`, minus the Kaspi workers:

1. `signal.NotifyContext` for `SIGINT`/`SIGTERM`.
2. Build config ([05](05-configuration.md)); init `slog` + OTel ([Observability](#observability)).
3. Build the DI container (lazy singletons): DB ‚Üí repos ‚Üí adapters (`chatwoot`, `anthropic`) ‚Üí `assistant` usecase ‚Üí HTTP handlers.
4. Start the chi HTTP server (read/write timeouts ~30s, body limit ~1 MB) in a goroutine.
5. On signal: `srv.Shutdown(ctx)` with a ~10s timeout to drain in-flight webhook requests. The `stop_grace_period: 25s` in compose gives headroom.

The brain has **no background workers** in v1 ‚Äî its only "loop" is the synchronous webhook handler. (A future callback sweeper, if added, follows the main repo's `ports/queue` ticker pattern.)

## Observability

Copy the main repo's setup wholesale:

- **Health:** `GET /health` and `GET /ready` ‚Üí `{"status":"ok"}` (used by the compose healthcheck and any proxy).
- **Metrics:** `GET /metrics` (Prometheus), bearer-token-gated by `METRICS_TOKEN`; RED metrics from chi middleware (reuse `xpayment/internal/ports/http/middleware/metrics.go`). Skip `/health`, `/ready`, `/metrics` from recording.
- **Logging:** `log/slog` JSON to stdout, wrapped by a `TraceHandler` that injects `trace_id`/`span_id` (reuse `xpayment/internal/infrastructure/telemetry`). Log per message: `chatID`, port-call latencies, LLM tokens, `confidence`, `escalate`, dropped `asset_refs`, leftover price tokens (see [02 ¬∑ post-processing](02-assistant-brain.md#post-processing-pipeline)).
- **Tracing:** OTel ‚Üí OTLP ‚Üí Jaeger, toggled by `OTEL_ENABLED`; `otelsql` wraps pgx. Reuse `telemetry/provider.go`. In local dev you can run the same `jaeger`/`jaeger-proxy` services from the main compose.

## Deployment

The brain is a single stateless service (its state is Postgres). **Recommended:** a simple single-container deploy on a VPS ‚Äî `docker compose up -d brain brain-postgres` with `restart: unless-stopped` behind the reverse proxy ([Backups & TLS](#backups--tls)). Upgrades = build the new image, `docker compose up -d --no-deps brain`; a few seconds of webhook unavailability is acceptable because Chatwoot retries and humans can reply manually meanwhile ([01 ¬∑ availability coupling](01-infrastructure.md#3-brain--chatwoot)).

**If zero-downtime is later required**, lift the main repo's blue-green pattern: the `app-blue`/`app-green` slots + HAProxy Runtime API + `xpayment/deploy.sh`. It is more than the brain needs at ~100 leads ‚Äî start simple.

CI is covered in [07-testing-and-evals.md](07-testing-and-evals.md#ci) (the repos currently have none).

## Backups & TLS

- **Backups:** Chatwoot's Postgres is now the **system of record** for every conversation, contact, and profile (Decision 1) ‚Äî back it up on a schedule (nightly `pg_dump` + offsite, test restores). The brain's own Postgres holds only authored config/KB/prices; back it up too, but losing it is recoverable from the admin UI / git, unlike Chatwoot's data. *(Cross-links [01 ¬∑ Operations](01-infrastructure.md#5-operations).)*
- **TLS / reverse proxy:** Chatwoot webhooks (and any public admin access) require **HTTPS**. Put Caddy or nginx in front of Chatwoot and the brain (Caddy gives automatic certs). The brain's `/v1/assistant/webhook/chatwoot` must be reachable over HTTPS in production; in local dev the tunnel provides this.
