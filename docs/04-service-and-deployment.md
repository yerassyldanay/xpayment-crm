How the standalone, **stateless** brain (Decisions 2, 12) is shaped, built, and run. Decisions and the repo-layout summary are in [README.md](README.md); env vars are in [05-configuration.md](05-configuration.md); the HTTP surface is in [06-api-and-contracts.md](06-api-and-contracts.md). ## Repo layout

Hexagonal, mirroring the main repo's package roles. The core (`domain`, `usecase/assistant`) has no infrastructure imports; adapters live under `infrastructure/`; transport under `ports/http/`. **No database, no migrations** (Decision 2) — config comes from the content repo.

```
xpayment-crm/
  cmd/main.go                 # load content snapshot, start HTTP server, graceful shutdown
  internal/
    domain/                   # Draft, Message, ChatID, Media, Snapshot … (stdlib only)
    usecase/assistant/        # HandleMessage(ctx, chatID, inbound) → Draft  + the ports (Decision 10)
      ports.go                # ContentSource, ChatwootReader, ChatwootWriter, Drafter, Prices, Catalog
      service.go              # the pipeline (02-assistant-brain.md)
      mocks/                  # generated by mockery
    infrastructure/
      content/                # ContentSource: load + validate xpayment-content → atomic Snapshot; git pull on reload
      chatwoot/               # ChatwootReader + ChatwootWriter (REST adapter, 06)
      anthropic/              # Drafter (LLM adapter, prompt caching)
      config/                 # Config struct + getEnv loader (05)
    ports/http/
      router.go               # chi router + middleware
      webhook.go              # POST /v1/assistant/webhook/chatwoot  (06)
      reload.go               # POST /v1/assistant/reload  (GitHub push → snapshot reload) (06)
  docs/                       # this set
  Dockerfile  docker-compose.yml  Makefile  .env.example  .mockery.yml

xpayment-content/             # the config — a separate repo (03); mounted or cloned at runtime
```

Reuse the package-role split from `xpayment/internal/` (`domain` → `usecase` → `infrastructure` / `ports`) and the lazy-init DI container from `xpayment/providers/container.go`.

## Dockerfile

Same multi-stage pattern as `xpayment/Dockerfile`: build a static binary on `golang:1.25-alpine`, run it on a minimal `alpine` with a non-root user. The runtime image needs **`git`** (the brain clones/pulls the content repo) but **no SQL** (no migrations).

```dockerfile
# Build stage
FROM golang:1.25-alpine AS build
WORKDIR /app
RUN apk add --no-cache git
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main ./cmd/main.go

# Deploy stage
FROM alpine:3.19 AS deploy
RUN apk add --no-cache tzdata ca-certificates wget git   # wget=healthcheck, git=pull content repo
RUN adduser -S -u 1001 appuser
ENV TZ=Asia/Almaty
WORKDIR /app
COPY --from=build /app/main ./main
EXPOSE 8080
USER appuser
ENTRYPOINT ["./main"]
```

The content repo reaches the brain in one of two ways (see [Content checkout & reload](#content-checkout--reload)): **mounted as a read-only volume**, or **cloned by the brain** from `CONTENT_REPO_URL`. Mount-mode needs no git in the image; clone-mode does — hence `git` is included.

## Local stack (docker-compose)

The main repo's compose runs *one* product; the brain's local stack stands up **three** (Chatwoot, Evolution, the brain) plus a tunnel so Chatwoot/GitHub can reach the brain's webhooks from your machine. The brain itself has **no database** — only the content repo (mounted below).

Chatwoot and Evolution publish their own official compose files — **compose them in rather than hand-rolling**. Sketch:

```yaml
# illustrative — pin image versions, set secrets via .env, verify upstream envs
services:
  # ── Chatwoot (hub) ── its own Postgres + Redis ───────────────────────────
  chatwoot-postgres: { image: postgres:16-alpine, volumes: [chatwoot_pg:/var/lib/postgresql/data] }
  chatwoot-redis:    { image: redis:7-alpine }
  chatwoot:
    image: chatwoot/chatwoot:latest        # pin a real tag
    depends_on: [chatwoot-postgres, chatwoot-redis]
    env_file: [.env]                        # POSTGRES_*, REDIS_URL, SECRET_KEY_BASE, FRONTEND_URL …
    ports: ["3000:3000"]

  # ── Evolution (WhatsApp gateway) ── its own Postgres + Redis ─────────────
  evolution-postgres: { image: postgres:16-alpine, volumes: [evo_pg:/var/lib/postgresql/data] }
  evolution-redis:    { image: redis:7-alpine }
  evolution:
    image: atendai/evolution-api:latest     # pin a real tag
    depends_on: [evolution-postgres, evolution-redis]
    env_file: [.env]                        # AUTHENTICATION_API_KEY, CHATWOOT_* integration …
    ports: ["8081:8080"]

  # ── Brain (this repo) ── stateless; reads the content repo, no DB ────────
  brain:
    build: { context: ., dockerfile: Dockerfile }
    env_file: [.env]                        # see 05-configuration.md
    volumes:
      - ../xpayment-content:/app/xpayment-content:ro   # mount-mode (CONTENT_REPO_PATH); or omit and set CONTENT_REPO_URL
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "-O", "/dev/null", "http://localhost:8080/health"]
      interval: 15s; timeout: 3s; start_period: 10s; retries: 3
    restart: unless-stopped
    stop_grace_period: 25s                   # graceful drain headroom (see Startup & shutdown)

  # ── Webhook tunnel (local dev only) ──────────────────────────────────────
  # Chatwoot's account webhook AND GitHub's reload webhook must reach the brain over public HTTPS.
  tunnel:
    image: cloudflare/cloudflared:latest     # or ngrok
    command: tunnel --url http://brain:8080
volumes: { chatwoot_pg: , evo_pg: }
```

Reuse from `xpayment/docker-compose.yml`: the `env_file: [.env]` wiring, the `wget /health` healthcheck, `restart: unless-stopped`, and `stop_grace_period: 25s`.

**Local-dev wiring order:** bring up Chatwoot → create the account + agent token → bring up Evolution and configure its Chatwoot integration ([01 · Evolution ↔ Chatwoot](01-infrastructure.md#1-evolution--chatwoot)) → mount/clone `xpayment-content` and bring up the brain → set the **tunnel URL** as Chatwoot's account webhook ([01 · The two webhook kinds](01-infrastructure.md#2-the-two-webhook-kinds--do-not-conflate)) and as the GitHub push webhook on `xpayment-content` ([06 · reload](06-api-and-contracts.md#github-reload-webhook)).

## Content checkout & reload

The brain's config is the `xpayment-content` repo loaded into an immutable in-memory `Snapshot` (file shapes + validation in [03](03-content-and-data.md)). Two delivery modes:

- **Mount mode (simplest):** mount a checkout at `CONTENT_REPO_PATH` (read-only volume). You update it by pulling on the host; the brain reloads on a `--reload` signal or poll.
- **Clone mode:** set `CONTENT_REPO_URL` (+ `CONTENT_REPO_BRANCH`); the brain `git clone`s on boot and `git pull`s on reload.

**Reload flow** (no restart, never serves a broken config):
1. A push to `xpayment-content`'s `main` hits `POST /v1/assistant/reload` (GitHub webhook, HMAC-verified with `RELOAD_WEBHOOK_SECRET` — [06](06-api-and-contracts.md#github-reload-webhook)).
2. The brain `git pull`s the checkout, builds a **new** `Snapshot`, and **validates** it ([03 · validate on load](03-content-and-data.md#validate-on-load-fail-loudly)).
3. On success it **atomically swaps** the snapshot pointer; on failure it keeps the old snapshot and logs the error.

So **publish = merge + push** ([08](08-admin-ui.md)); the bot picks up good changes within seconds and rejects bad ones.

## Startup & shutdown

Mirror `xpayment/cmd/main.go`, minus the Kaspi workers and the DB:

1. `signal.NotifyContext` for `SIGINT`/`SIGTERM`.
2. Build config ([05](05-configuration.md)); init `slog` + OTel ([Observability](#observability)).
3. **Load the content snapshot** (mount or clone), validate it, and refuse to boot if invalid.
4. Build the DI container (lazy singletons): `ContentSource` → adapters (`chatwoot`, `anthropic`) → `assistant` usecase → HTTP handlers (webhook + reload).
5. Start the chi HTTP server (read/write timeouts ~30s, body limit ~1 MB) in a goroutine.
6. On signal: `srv.Shutdown(ctx)` with a ~10s timeout to drain in-flight webhook requests. The `stop_grace_period: 25s` in compose gives headroom.

The brain has **no background workers** in v1 — its loops are the synchronous webhook handler and (optionally) a content-poll. (A future callback sweeper, if added, follows the main repo's `ports/queue` ticker pattern.)

## Observability

Copy the main repo's setup wholesale:

- **Health:** `GET /health` and `GET /ready` → `{"status":"ok"}` (used by the compose healthcheck and any proxy).
- **Metrics:** `GET /metrics` (Prometheus), bearer-token-gated by `METRICS_TOKEN`; RED metrics from chi middleware (reuse `xpayment/internal/ports/http/middleware/metrics.go`). Skip `/health`, `/ready`, `/metrics` from recording.
- **Logging:** `log/slog` JSON to stdout, wrapped by a `TraceHandler` that injects `trace_id`/`span_id` (reuse `xpayment/internal/infrastructure/telemetry`). Log per message: `chatID`, port-call latencies, LLM tokens, `confidence`, `escalate`, dropped `asset_refs`, leftover price tokens (see [02 · post-processing](02-assistant-brain.md#post-processing-pipeline)); and per reload: commit sha + validation result.
- **Tracing:** OTel → OTLP → Jaeger, toggled by `OTEL_ENABLED`. Reuse `telemetry/provider.go`. In local dev you can run the same `jaeger`/`jaeger-proxy` services from the main compose.

## Deployment

The brain is a single **stateless** service (its state is Chatwoot + the content repo). **Recommended:** a simple single-container deploy on a VPS — `docker compose up -d brain` with `restart: unless-stopped` behind the reverse proxy ([Backups & TLS](#backups--tls)). Upgrades = build the new image, `docker compose up -d --no-deps brain`; a few seconds of webhook unavailability is acceptable because Chatwoot retries and humans can reply manually meanwhile ([01 · availability coupling](01-infrastructure.md#3-brain--chatwoot)).

**If zero-downtime is later required**, lift the main repo's blue-green pattern: `app-blue`/`app-green` + HAProxy Runtime API + `xpayment/deploy.sh`. It is more than the brain needs at ~100 leads — start simple.

CI is covered in [07-testing-and-evals.md](07-testing-and-evals.md#ci) (the repos currently have none).

## Backups & TLS

- **Backups:** Chatwoot's Postgres is the **system of record** for every conversation, contact, and profile (Decision 1) — back it up on a schedule (nightly `pg_dump` + offsite, test restores). The brain itself has **nothing to back up** — it is stateless; its config lives in the `xpayment-content` git repo (and on GitHub). *(Cross-links [01 · Operations](01-infrastructure.md#5-operations).)*
- **TLS / reverse proxy:** Chatwoot's account webhook and GitHub's reload webhook both require **HTTPS** to the brain. Put Caddy or nginx in front of Chatwoot and the brain (Caddy gives automatic certs). The brain's `/v1/assistant/*` endpoints must be reachable over HTTPS in production; in local dev the tunnel provides this.
